## Control Plane Position

This contract is issued by `myworld` as control plane. CapabilityOS owns the
implementation. Hive Mind and MemoryOS are read-only reference surfaces for
ASC-0002; they are not implementation targets in this slice.

## Goal

CapabilityOS is currently 0% code. Hivemind has a `memory_bridge.py` pattern that shows how an OS hook can be optional and non-blocking. ASC-0002 establishes the analogous surface for CapabilityOS:

1. CapabilityOS exposes a callable contract surface (CLI or Python entrypoint) that takes a task description and returns a recommendation payload.
2. hivemind's route phase can call this surface in the same non-blocking, graceful-degrade pattern as `memory_bridge.py` (continue if absent, record reason).
3. The recommendation payload is **minimal but real** — not a stub that returns nothing; it must contain at least one actionable field (recommended_capability id, fallback id, confidence).

ASC-0002 explicitly does **not** require:
- the full Capability Ontology JSON schema to be finalized
- the MVP Design-to-Code workflow to be implemented
- Surfer/Discriminator architecture
- any data ingestion pipeline

The goal is to make the third leg of the AIOS *real* — even if it returns a hand-curated recommendation list at first — so that ASC-0001-style dogfood loops can include CapabilityOS.

## Design Answers

**Q1 — Surface shape:** V1 uses both a CLI and Python entrypoint. The CLI is
`python -m capabilityos.cli ...`; the Python surface is
`capabilityos.load_catalog()`, `capabilityos.recommend()`, and
`capabilityos.audit_catalog()`. This mirrors the MemoryOS pattern of a JSON CLI
with importable helpers, without requiring Hive integration in the same slice.

**Q2 — Recommendation payload schema:** V1 returns:

- `task`
- `recommendations[]`
- per recommendation: capability card fields, `score`, `reason_codes`,
  `fallback_ids`, `risk_notes`
- `total_candidates`

This is slightly richer than the suggested floor because route scoring and
fallbacks are needed for useful AIOS dispatch decisions, but it remains
read-only and local.

**Q3 — Data source for V1:** V1 uses a hand-curated JSON fixture at
`CapabilityOS/tests/fixtures/capabilities.json`. It is committed as a small
synthetic catalog, not generated from Hive run history and not stored in
`CapabilityOS/data/`, because `data/` should remain local/generated by default.

**Q4 — Verification gate:** CapabilityOS-side tests and CLI smokes prove the
surface works. Codex recommends moving the Hive bridge/degrade test to ASC-0003
instead of widening ASC-0002, because adding `hivemind/capability_bridge.py`
is a second repo implementation slice.

**Q5 — Catalog write authority:** V1 is read-only. No code writes to the
catalog. Catalog write authority, ingestion, or learned capability performance
memory requires a later contract.

## Counter-Proposal

Do not include Hive Mind bridge implementation in ASC-0002. The stronger split
is:

- ASC-0002: CapabilityOS owns its first local recommendation-only executable
  surface.
- ASC-0003: Hive Mind consumes CapabilityOS recommendations through an optional
  nonblocking `capability_bridge.py` pattern analogous to `memory_bridge.py`.

Reason: ASC-0002 is the first code in a previously docs-only repo. Mixing
CapabilityOS package creation and Hive route integration in one contract makes
scope, ownership, and verification less sharp. The implemented CLI now gives
ASC-0003 a concrete target.

  `risk_notes`

  `recommendation_only: true`

## Stop Conditions

- `ontology_scope_creep`: full ontology schema demanded by V1.
- `mvp_vertical_demand`: Design-to-Code workflow demanded by V1.
- `hivemind_bridge_creep`: Hive route integration demanded before the
  CapabilityOS CLI target exists.
- `schema_drift_with_existing_memory_bridge`: future Hive bridge diverges from
  MemoryOS bridge's optional/nonblocking pattern without justification.
- `catalog_write_creep`: V1 demands write API, ingestion, or learned
  performance mutation.
- `execution_creep`: V1 executes tools, provider CLIs, MCP bindings, or network
  calls.
